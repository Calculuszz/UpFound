# UpFound — Technical Specification

เอกสารอธิบายสถาปัตยกรรม สัญญาข้อมูล (data contract) กระบวนการประมวลผล และเทคโนโลยีที่ใช้ของระบบ UpFound

หลักการออกแบบหลัก: แยกส่วนประกอบทั้งหมดออกจากกันผ่าน **Event Contract** เพื่อให้แหล่งสัญญาณภาพ (วิดีโอที่บันทึกไว้ อุปกรณ์ประมวลผลที่ขอบเครือข่าย หรือกล้องวงจรปิดแบบสด) สามารถสลับเปลี่ยนกันได้โดยส่วนปลายทางไม่ต้องแก้ไข

---

## 1. ภาพรวมสถาปัตยกรรม

ระบบประกอบด้วยสามส่วนหลัก โดยมี Event Contract เป็นเส้นแบ่งความรับผิดชอบ

| ส่วนประกอบ | ขอบเขตงาน |
|---|---|
| Edge AI | รับสัญญาณภาพจากแหล่งต่าง ๆ ตรวจจับวัตถุ เข้ารหัสคุณลักษณะ และส่งออกเป็นเหตุการณ์ตามสัญญาข้อมูล |
| Workflow AI | รับเหตุการณ์ จับคู่แบบไล่ระดับ สร้างเส้นทางของวัตถุ ประเมินความมั่นใจ พิสูจน์ความเป็นเจ้าของ และออกบัตรรับของ |
| Web Application | ส่วนติดต่อผู้ใช้: แบบฟอร์มแจ้ง การแสดงรายการและหลักฐาน การพิสูจน์ตัวตน และการรับบัตร QR |

```
[วิดีโอ/Edge/CCTV] --produce--> | Event Contract | --consume--> [Workflow AI] <--> [Web Application]
```

## 2. การไหลของข้อมูล

**เส้นทางตรวจจับ (เขียนข้อมูล):** แหล่งสัญญาณ → ตรวจจับวัตถุที่ถูกทิ้ง → ตัดภาพและเข้ารหัสคุณลักษณะ → สร้างเหตุการณ์ → ส่งเข้าคิวข้อความ → จัดเก็บลงบันทึกและฐานข้อมูลเวกเตอร์

**เส้นทางตามหา (อ่านข้อมูล):** คำขอจากผู้ใช้ → จับคู่กับฐานข้อมูลเวกเตอร์และบันทึก → สร้างเส้นทางของวัตถุและรายการที่เป็นไปได้ → ประเมินความมั่นใจ → พิสูจน์ความเป็นเจ้าของ → ออกบัตรรับของ

ทั้งสองเส้นทางมาบรรจบกันที่ขั้นตอนการจับคู่ โดยเส้นทางตรวจจับทำหน้าที่เติมข้อมูลในคลัง และเส้นทางตามหาทำหน้าที่ค้นจากคลัง

## 3. Event Contract

สัญญาข้อมูลกลางที่ทุกส่วนยึดถือร่วมกัน (ตรวจสอบความถูกต้องด้วย Pydantic และระบุ `schema_version`)

```json
{
  "schema_version": "1.0",
  "model_version":  "yolo11n_clip-vitb32",
  "event_id":       "a1b2c3d4e5f6",
  "source":         "replay | edge | cctv",
  "camera_id":      "cam-03",
  "zone":           "fl2-zoneA",
  "capture_ts":     "2026-01-01T14:32:05.120Z",
  "detect_type":    "abandoned_object",
  "object_class":   "backpack",
  "track_id":       7,
  "bbox":           [320, 180, 90, 140],
  "crop_ref":       "s3://.../cam03_1432_obj7.jpg",
  "embedding":      [0.12, -0.88, "... 512 มิติ, normalize แบบ L2"]
}
```

**ข้อกำหนดสำคัญสองข้อ:**

1. `model_version` — เวกเตอร์คุณลักษณะที่สร้างจากโมเดลต่างเวอร์ชันกันไม่สามารถนำมาเทียบกันได้ การเปลี่ยนโมเดลต้องสร้างเวกเตอร์ใหม่ทั้งคลัง
2. `capture_ts` ต้องเป็นเวลาที่บันทึกภาพจริง ไม่ใช่เวลาที่ข้อมูลเข้าสู่ระบบ มิฉะนั้นการกรองตามพื้นที่และเวลาในขั้นจับคู่จะคลาดเคลื่อน

## 4. กระบวนการแปลงวิดีโอเป็นเหตุการณ์

แหล่งสัญญาณประเภทวิดีโอที่บันทึกไว้มีคุณสมบัติ deterministic (ลำดับเฟรมแปลงเป็นเวลาได้แน่นอน) จึงเหมาะเป็นทั้งแหล่งทดสอบและแหล่งสำรอง

ขั้นตอนการประมวลผล:

1. ถอดรหัสและสุ่มเฟรม โดยคำนวณเวลาจากลำดับเฟรมและอัตราเฟรม ประมวลผลทุก N เฟรม
2. ตรวจจับและติดตามวัตถุ โดยเลือกเฉพาะประเภทกระเป๋าและสัมภาระ พร้อมกำหนดรหัสติดตาม
3. ตรรกะการทิ้ง — หากตำแหน่งของวัตถุเคลื่อนเกินเกณฑ์จะรีเซ็ตตัวจับเวลา หากนิ่งต่อเนื่องเกินกำหนดจะถือว่าเป็นวัตถุที่ถูกทิ้ง
4. สร้างเหตุการณ์เพียงครั้งเดียวต่อหนึ่งรหัสติดตาม เพื่อป้องกันการสร้างเหตุการณ์ซ้ำ
5. ตัดภาพตามกรอบและเข้ารหัสคุณลักษณะ
6. ประกอบเป็นเหตุการณ์ตามสัญญาข้อมูลและส่งเข้าคิว

## 5. การจับคู่แบบไล่ระดับ

จับคู่จากเงื่อนไขที่ประมวลผลถูกที่สุดไปแพงที่สุด เพื่อตัดรายการที่ไม่เกี่ยวข้องออกก่อน

1. กรองตามพื้นที่และช่วงเวลา (ตัดรายการที่ไม่เกี่ยวข้องออกเป็นส่วนใหญ่)
2. จับคู่ตามคุณลักษณะ (ประเภท สี ยี่ห้อ) จากข้อมูลในแบบฟอร์ม
3. จัดอันดับใหม่ด้วยความคล้ายของภาพ ซึ่งรองรับทั้งการเทียบภาพต่อภาพและการเทียบข้อความต่อภาพ ใช้เป็นตัวจัดอันดับเสริม ไม่ใช่ตัวตัดสินหลัก

## 6. การประเมินความมั่นใจและการปฏิเสธ

ค่าความมั่นใจต้องผ่านการปรับเทียบ และระบบต้องสามารถปฏิเสธได้เมื่อข้อมูลไม่ชัดเจน

| ระดับความมั่นใจ | การดำเนินการ |
|---|---|
| สูง | เสนอรายการที่เป็นไปได้สูงสุดสามอันดับพร้อมหลักฐาน |
| ปานกลาง | ส่งให้เจ้าหน้าที่ตรวจสอบ |
| ต่ำ | แจ้งว่ายังไม่พบ และแจ้งเตือนเมื่อพบในภายหลัง |

การออกแบบให้ระบบยอมรับว่ายังไม่พบ ช่วยลดการจับคู่ผิดพลาดในกรณีที่วัตถุมีลักษณะคล้ายกันจำนวนมาก

## 7. การพิสูจน์ความเป็นเจ้าของ

ระบบไม่สามารถมองเห็นสิ่งของภายในได้ จึงตั้งคำถามเฉพาะสิ่งที่ภาพจากกล้องสามารถยืนยันได้ แล้วนำคำตอบไปเทียบกับเฟรมที่บันทึกผู้เป็นเจ้าของไว้ เช่น พื้นที่ที่อยู่ จำนวนผู้ที่มาด้วย สีเสื้อ ทิศทางการเดินเข้า และเวลาโดยประมาณที่วางของ

หากผ่านการพิสูจน์ ระบบจะออกบัตรรับของดิจิทัลแบบชั่วคราว หากไม่ผ่านจะส่งเข้ากระบวนการตรวจสอบโดยเจ้าหน้าที่ ส่วนการยืนยันสิ่งของภายในจริงให้เจ้าหน้าที่ดำเนินการขณะส่งมอบ

### 7.1 การเทียบใบหน้าแบบเฉพาะกรณี (Ephemeral Face Match)

ชั้นการพิสูจน์เสริมที่เทียบใบหน้าของผู้มารับกับใบหน้าของผู้ที่นำวัตถุมาวางในเฟรม เฉพาะภายในกรณีนั้น

- ขั้นตอน: ตรวจจับใบหน้า ปรับแนว เข้ารหัส และเทียบแบบหนึ่งต่อหนึ่งด้วยค่าความคล้าย แล้วคืนผลผ่านหรือไม่ผ่าน
- ไม่มีฐานข้อมูลใบหน้า ไม่ระบุตัวบุคคล และไม่ค้นจากคลังใบหน้า เป็นการเทียบสองภาพต่อกันเท่านั้น
- เวกเตอร์ใบหน้าทั้งสองอยู่ในหน่วยความจำระหว่างการพิสูจน์เท่านั้น และถูกลบทันทีเมื่อสิ้นสุด ไม่มีการบันทึกลงที่จัดเก็บข้อมูล
- ใช้ประกอบกับคำถามที่ตรวจสอบกับภาพได้ ไม่ใช้แทนที่ เพื่อรองรับกรณีที่ใบหน้าถูกบังหรือมุมกล้องไม่เหมาะสม

## 8. Technology Stack

| ส่วน | เทคโนโลยี | เหตุผล |
|---|---|---|
| ภาษา | Python | — |
| ถอดรหัสวิดีโอ | OpenCV (ปรับเป็น PyAV/decord ได้เมื่อต้องการประสิทธิภาพ) | รองรับไฟล์วิดีโอทั่วไป |
| ตรวจจับและติดตามวัตถุ | Ultralytics YOLO v26 (ByteTrack) | ครอบคลุมประเภทกระเป๋าและสัมภาระ |
| เข้ารหัสคุณลักษณะภาพ | OpenCLIP ViT-B/32 (512 มิติ) | รองรับการเทียบภาพต่อภาพและข้อความต่อภาพ |
| การเทียบใบหน้า (เฉพาะกรณี) | InsightFace / ArcFace ใช้แบบหนึ่งต่อหนึ่ง | เทียบสองภาพต่อกัน ไม่ค้นจากคลัง ไม่จัดเก็บเวกเตอร์ |
| สัญญาข้อมูล | Pydantic | ตรวจสอบความถูกต้องและกำหนดเวอร์ชัน |
| การรับส่งข้อมูล | คิวข้อความ (Redis Stream / Kafka) | แยกผู้ผลิตและผู้บริโภคข้อมูล และรองรับภาระสูง |
| ฐานข้อมูลเวกเตอร์ | Qdrant / pgvector | ค้นด้วยความคล้าย |
| การจัดรูปแบบข้อมูล | protobuf / avro | เหมาะกับเวกเตอร์ขนาดใหญ่กว่ารูปแบบข้อความ |

## 9. การรองรับหลายแหล่งสัญญาณและความทนทาน

- ระบบรองรับแหล่งสัญญาณสามประเภท (วิดีโอที่บันทึกไว้ อุปกรณ์ประมวลผลที่ขอบเครือข่าย และกล้องวงจรปิดแบบสด) ผ่านสัญญาข้อมูลเดียวกัน ส่วนปลายทางจึงไม่จำเป็นต้องทราบที่มาของข้อมูล
- ความสามารถนี้รองรับการสลับแหล่งสัญญาณได้อย่างต่อเนื่อง เพื่อความยืดหยุ่นในการทดสอบและการให้บริการ รวมถึงการลดระดับการทำงานอย่างนุ่มนวลเมื่อสัญญาณสดขัดข้อง
- สามารถประมวลผลภายในเครือข่ายท้องถิ่นได้โดยไม่ต้องพึ่งบริการคลาวด์

## 10. ความเป็นส่วนตัวและความปลอดภัย

- ประมวลผลที่ขอบเครือข่าย วิดีโอต้นฉบับไม่ออกจากอุปกรณ์ ส่งออกเฉพาะข้อมูลกำกับ ภาพที่ตัดแล้ว และเวกเตอร์คุณลักษณะ
- เบลอใบหน้าเป็นค่าตั้งต้น
- กำหนดระยะเก็บข้อมูลสั้น ลบข้อมูลต้นฉบับอัตโนมัติ เก็บเฉพาะกรณีที่ยังดำเนินการอยู่
- มีการควบคุมสิทธิ์การเข้าถึงและบันทึกการตรวจสอบ
- การเทียบใบหน้าใช้แบบเฉพาะกรณีเท่านั้น ไม่มีฐานข้อมูลใบหน้า ไม่ทำการจดจำหรือระบุตัวบุคคล และไม่จัดเก็บเวกเตอร์ใบหน้า

## 11. การกำหนดเวอร์ชันของ Schema และ Model

ทุกเหตุการณ์ระบุ `schema_version` และ `model_version` การเปลี่ยนแปลงโมเดลหรือโครงสร้างข้อมูลต้องมีแนวทางการย้ายข้อมูลที่ชัดเจน รวมถึงการสร้างเวกเตอร์คุณลักษณะใหม่ทั้งคลังเมื่อจำเป็น

## 12. ประเด็นทางเทคนิคที่ต้องตัดสินใจ

- อุปกรณ์ตรวจจับที่ขอบเครือข่ายจะจัดเก็บเวกเตอร์คุณลักษณะของวัตถุตั้งแต่ขั้นตรวจจับหรือไม่ ซึ่งส่งผลต่อความจำเป็นของการค้นย้อนหลังในวิดีโอ
- การเลือกโมเดลเข้ารหัสคุณลักษณะสำหรับการเทียบข้ามลักษณะภาพ (ภาพจากอุปกรณ์ส่วนตัวที่คมชัด เทียบกับภาพจากกล้องวงจรปิดที่ความละเอียดต่ำกว่า)
- ความสามารถในการดึงสัญญาณจากกล้องวงจรปิดผ่านโปรโตคอลมาตรฐานและคุณภาพเครือข่ายในพื้นที่ติดตั้ง
- แนวทางการย้ายข้อมูลเมื่อมีการเปลี่ยนเวอร์ชันของ schema หรือ model
